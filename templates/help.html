Help

Lodge Office is a local-first web app for lodge administration: members, treasury, reporting, and optional catering workflows. Use the menu (☰) to pick a workspace (for example Lodge · Treasurer). The full lodge accounts bar appears for Lodge · Treasurer or Lodge · Admin. Lodge · Secretary shows a shorter bar (Members, Catering, Help, Settings) using the same underlying data for now.

Overview

The app runs in your web browser. On a Windows PC it is usually started with start.bat, which creates a virtual environment if needed and opens the site at http://127.0.0.1:5000 (or as configured). The live data is stored in a SQLite database file on disk (often named LodgeOffice.db in the app folder, or paths set in config.local / environment variables).

After sign-in, choosing Lodge · Treasurer or Lodge · Admin opens Treasurer home and the full accounts navigation (Members, Bank, Cash, Statement, catering, and so on). Lodge · Secretary opens Members first and shows Members, Catering, Help, and Settings (same data as the treasurer views for those areas). Forms and Settings are available to every signed-in user for now.

Sign-in, roles, and permissions

You sign in with an email and password. Your administrator creates portal users under Settings → Portal users. Each person gets a directory role label (for example Treasurer, Secretary) for your records, but what they actually see is set per person: which menu (☰) workspaces they may use (lodge vs chapter, and office such as Treasurer or Secretary), and optional admin capabilities (managing portal users or the table editor).

The full treasurer navigation bar appears when the active workspace is Lodge · Treasurer or Lodge · Admin. Lodge · Secretary uses the shorter bar described above. Fine-grained per-item toggles for the treasurer bar are not implemented.

Where your data lives

  • Primary database: the working SQLite file the app reads and writes during normal use.
  • Mirrored backup: after successful saves, the app can copy the database to a backup file (see Settings). Use this for recovery if the primary file is damaged.
  • Excel workbook: some imports and seeds may reference a workbook next to the app; server deployments may not have one—use in-app screens instead.

For deployment, Docker, and server paths, your operator should follow the project runbooks shipped with the repository (for example docs/Runbook.md and hosting notes).

Help
This page.
Settings
Lodge display name, lodge meeting cycle (repeating pattern for the five meetings), virtual account mapping, opening balances, backup folder, database download/upload, and portal admin links.

The grid icon (waffle) in the header opens the same destinations in a compact menu.

Treasurer workflows

Members and dues

Record subscription and dining dues per member for the reporting period. Posts from this area feed reporting and cash expectations.

Bank

Import or enter bank lines, assign ledger categories, and reconcile splits. Large CSV uploads may be limited by size; fix encoding errors if the app reports them.

Cash

Cash is organised by the same meeting keys as in Settings (September, November, January, March, May). Link deposits to the right meeting and use settlements to tie cash to bank when you bank the takings.

Statement and sub accounts

Use these for period reporting and for trustees’ views of designated funds. Exact layout depends on your lodge’s configuration.

Settings

Lodge name on reports: shown on the statement and similar outputs; leave blank to use the install default.

Lodge meeting cycle: for each of the five meetings, set the calendar month, weekday, and which week of the month (for example 3rd Saturday). The app stores the next occurrence on or after today as the meeting date. Cash and catering use these dates.

Virtual account mapping: choose which internal balance-sheet account each bank category feeds.

Opening balances: figures that roll into the new year as configured for your reporting period.

Backup: set a backup folder the app can write to; use Back up now to mirror the database. Restore replaces the live file from the mirror (destructive—confirm before use).

Database file: download a copy for cold storage, or upload a replacement file only when you intend to replace the entire live database (your administrator should coordinate this).

People granted the relevant admin capabilities see Portal users and/or Table admin on the Settings page (configured per person under Portal users).

Catering

Catering menu and prices is the master list of dishes and supplements (editable when your caterer changes the menu). New meal booking creates an event; on setup you can apply dishes from the master list, adjust wording and supplements for that night, and copy a public link.

Anyone with the link can submit choices without signing in to the rest of Lodge Office. Treat the link like a secret—regenerate it in setup if it is leaked.

Optionally link a booking to a lodge meeting from the list (dates come from Settings). Responses are listed under each booking for the Secretary or catering officer.

Backups and safety

  • Keep the mirrored backup path on reliable storage (local Documents, OneDrive, or server disk as appropriate).
  • Before major changes (restore, database upload), take an extra manual copy of the database file.
  • Only one instance of the app should hold the runtime lock against the same live database—do not run two copies against one file.
  • For production servers, use HTTPS and normal server hardening; the public meal link is unguessable but should not be posted publicly if you want to limit who books.
